## 1. Project Overview and Goals

DietBite is an Indian nutrition intelligence platform built primarily for **dietitians, nutritionists, and doctors**, with secondary support for **health-conscious individual users**.

The platform specializes in Indian foods and nutrition science, using data structured in the spirit of:
- **IFCT** (Indian Food Composition Table, 2017)
- **NIN** (National Institute of Nutrition) guidelines

### Core problem being solved
Dietitians and nutrition professionals currently spend significant time manually calculating nutrients, planning meals, and analyzing Indian foods — most global nutrition apps are built around Western food databases and don't serve Indian clinical workflows well.

DietBite automates:
- Nutrient calculations (per food and per meal)
- Meal planning workflows (structured, multi-day, multi-patient)
- Food search and analysis (Indian-first database)
- Patient progress tracking (weight, BMI, history)

### Target users and roles
| Role | Description | Core capabilities |
|---|---|---|
| **Dietitian** | Primary user | Patient management, meal plan building, food search, nutrient calculator |
| **User** | Health-conscious individual | Food search, nutrient calculator, personal tracking (no patient access) |
| **Admin** | Platform operator | User management, role/status control, oversight of patients (read access) |

### Product vision
A modern, Indian-first nutrition platform combining scientific accuracy, professional clinical workflows, a strong Indian food database, and a scalable architecture that can grow toward AI-assisted features in the future — without needing to be re-architected.

## 4. Current Architecture

### Backend layering
```
Router (HTTP layer, FastAPI)
   ↓ depends on
Service (business logic, plain functions taking a db Session)
   ↓ queries
Model (SQLAlchemy ORM, maps to Postgres tables)
```

Schemas (Pydantic) sit alongside this and are used purely for request validation and response shaping — they never contain business logic.

### Why this layering
- Routers stay thin — they only handle HTTP concerns (status codes, dependency injection) and delegate everything else.
- Services are framework-agnostic and testable in isolation.
- This is the same shape we'll need if/when a mobile app or CLI tool needs to reuse the same business logic without going through HTTP.

### Frontend layering
```
Page (route-level component, owns data fetching + state)
   ↓ uses
Service (api/*Service.js — wraps axios calls, no React)
   ↓ calls
Axios instance (api/axios.js — base URL, auth interceptor, refresh logic)
```

Reusable presentational pieces live in `components/`, organized by domain (food, calculator, patients, mealplans) plus a generic `ui/` folder for primitives used everywhere (Button, Input, Card, Alert).

### Data flow for a typical feature (example: Meal Plan)
1. `MealPlanDetailPage` fetches plan detail via `mealPlanService.getById`
2. Backend computes nutrient totals **fresh on every read** (not stored/cached) — `services/meal_plan.py::get_plan_detail` scales each food's per-100g nutrients by `quantity_g / 100` and sums them
3. This same scaling math is duplicated conceptually in three places: `services/food.py::calculate_nutrition` (calculator), `services/meal_plan.py` (meal plans), and nowhere else — **this duplication is a known, accepted simplification for now** (see Section 11)

## 5. Authentication and Role System

### Auth mechanism
- JWT-based, with **separate access and refresh tokens**
- Access token: short-lived (default 30 min), used for all authenticated requests via `Authorization: Bearer <token>`
- Refresh token: longer-lived (default 7 days), used only to mint new access tokens via `POST /auth/refresh`
- Passwords hashed with **bcrypt** via Passlib — never stored or logged in plaintext
- Frontend stores both tokens in `localStorage` and auto-attaches/auto-refreshes them via an Axios interceptor (`api/axios.js`)

### Roles
Defined as a Postgres enum (`UserRole`) on the `users` table: `admin`, `dietitian`, `user`.

| Capability | admin | dietitian | user |
|---|---|---|---|
| Manage all users (roles, activation, deletion) | ✅ | ❌ | ❌ |
| Create/manage patients | ✅ (oversight) | ✅ (own only) | ❌ |
| Build meal plans | ✅ (oversight) | ✅ (own patients only) | ❌ |
| Food search | ✅ | ✅ | ✅ |
| Nutrient calculator | ✅ | ✅ | ✅ |
| Own profile management | ✅ | ✅ | ✅ |

### Enforcement points
- **Backend**: `app/core/dependencies.py` exposes `get_current_user` (any authenticated user) and `require_role(*roles)` (role allow-list factory). Every protected router uses one of these as a FastAPI `Depends`.
- **Ownership checks**: Beyond role checks, patient and meal-plan endpoints verify `dietitian_id` matches the requesting user (see `services/patient.py::get_patient_by_id` and `services/meal_plan.py::_verify_patient_ownership`). Admins currently bypass this in some places for oversight — **this is intentional but should be revisited before going to production** (see Section 9).
- **Frontend**: `routes/ProtectedRoute.jsx` (any logged-in user) and `routes/RoleRoute.jsx` (specific roles) wrap routes in `App.jsx`. The sidebar (`AppLayout.jsx`) also conditionally renders nav links per role, but this is a UX convenience only — **it is not a security boundary**. All real enforcement happens server-side.

## 6. Database Summary

### Tables (as of Milestone 10)

| Table | Purpose | Key relationships |
|---|---|---|
| `users` | Auth + role identity | — |
| `food_categories` | Grouping for foods (e.g. "Pulses & Legumes") | has many `foods` |
| `foods` | Core Indian food database entries | belongs to `food_categories`, has many `food_nutrients` |
| `food_nutrients` | Per-food nutrient values (per 100g) | belongs to `foods` |
| `patients` | Dietitian's patient records | belongs to `users` (dietitian), has many `patient_measurements`, `meal_plans` |
| `patient_measurements` | Weight/BMI history over time | belongs to `patients` |
| `meal_plans` | A named, dated plan assigned to a patient | belongs to `patients` and `users` (dietitian), has many `meal_plan_days` |
| `meal_plan_days` | A single day within a plan | belongs to `meal_plans`, has many `meal_plan_items` |
| `meal_plan_items` | A specific food + quantity + meal slot within a day | belongs to `meal_plan_days` and `foods` |

### Key design decisions
- **Nutrients are EAV-style** (`food_nutrients` is a name/value/unit table, not fixed columns). This was chosen so new nutrients can be added without schema migrations — a real need given IFCT/NIN datasets have 50+ tracked nutrients we haven't all seeded yet.
- **Nutrition totals are never stored** — they're computed at read-time by scaling per-100g values. This guarantees consistency (no stale cached totals) at the cost of recomputing on every request. Acceptable at current scale; would need caching if food/item counts grow very large.
- **Patients store current stats AND have a measurement history** — `patients.current_weight_kg` is a denormalized "latest" pointer for fast list/card display, while `patient_measurements` is the append-only source of truth for progress charts.
- All primary keys are **UUIDs**, not auto-incrementing integers — chosen to avoid exposing sequential IDs and to make future multi-tenant or distributed scenarios easier.

### Migrations
Managed exclusively through **Alembic**. `Base.metadata.create_all()` was removed from `main.py` after Milestone 4 — **all schema changes must go through an Alembic revision**, never direct table edits.

## 7. Key Backend APIs/Modules

All routes are prefixed with `/api/v1`. Full interactive documentation is always available at `http://localhost:8000/docs` (Swagger UI).

| Module | Base path | Notable endpoints |
|---|---|---|
| **Auth** | `/auth` | `POST /signup`, `POST /login`, `POST /refresh`, `GET /me` |
| **Users** | `/users` | `GET /me`, `PATCH /me`, `PATCH /me/password` |
| **Admin** | `/admin` | `GET /users`, `GET /users/{id}`, `PATCH /users/{id}`, `DELETE /users/{id}` |
| **Foods** | `/foods` | `GET /search` (filters: `q`, `category`, `food_type`, `region`), `GET /categories`, `GET /{id}`, `GET /{id}/nutrients`, `POST /calculate` |
| **Patients** | `/patients` | `POST ""`, `GET ""`, `GET /{id}`, `PATCH /{id}`, `DELETE /{id}`, `POST /{id}/measurements`, `GET /{id}/measurements` |
| **Meal Plans** | `/meal-plans` | `POST ""`, `GET /by-patient/{patient_id}`, `GET /{id}`, `PATCH /{id}`, `DELETE /{id}`, `POST /{id}/days`, `DELETE /days/{day_id}`, `POST /days/{day_id}/items`, `DELETE /items/{item_id}` |

### Cross-cutting concerns
- **CORS**: currently locked to `http://localhost:5173` (the Vite dev server) in `main.py`. **Must be updated before deployment** to the production frontend origin.
- **Error handling**: services raise `fastapi.HTTPException` directly with appropriate status codes (`400`, `401`, `403`, `404`); there is no global exception handler middleware yet.

| `/patients` | dietitian, admin | `PatientListPage` |
| `/patients/:id` | dietitian, admin | `PatientDetailPage` (includes meal plan list) |
| `/meal-plans/:id` | dietitian, admin | `MealPlanDetailPage` |
| `/admin/users` | admin only | `AdminUsersPage` |

### State management approach
No external state library (Redux, Zustand, etc.) is used. State is managed via:
- **`AuthContext`** — the only truly global state (current user, login/signup/logout functions)
- **Local component state** (`useState`) for everything else — each page owns its own data fetching and loading/error state
- This is intentional for the current scale. If cross-page state sharing needs grow significantly, this should be revisited rather than organically growing prop-drilling.

### Styling approach
Tailwind utility classes directly in JSX, no CSS modules or styled-components. A consistent color/spacing vocabulary has emerged organically (green as primary/brand color, gray-50/100 backgrounds, rounded-xl cards) but **has not yet been formalized into a design system** — this is the explicit subject of the upcoming UI/UX overhaul.

## 10. Current Limitations

These are known, accepted gaps at this stage of the project — not bugs, but scope boundaries:

- **No AI features yet** — no chatbot, OCR, image recognition, or recommendation engine. Explicitly deferred per original MVP scope.
- **No mobile app** — web-only (responsive behavior not yet verified/optimized either — part of the upcoming UI/UX overhaul).
- **No automated tests** — no unit, integration, or E2E test suite exists yet. All verification so far has been manual (Swagger UI + browser click-through).
- **No report/PDF export** — dietitians cannot yet export a meal plan or patient report as a shareable document.
- **No search tolerance** — food search is a simple `LIKE` query; no typo tolerance, fuzzy matching, or proper Hinglish transliteration support yet (raw Hindi strings are seeded as a separate field, which is a workaround, not the real solution).
- **No caching layer** — nutrient totals are recomputed on every request. Fine at current data volume.
- **Admin oversight bypass on patients/plans is not fully scoped** — admins can read any patient/plan via the `dietitian_only` role guard (which includes admin), but the exact intended admin permissions (read-only? full edit? audit logging?) haven't been explicitly decided.
- **No rate limiting, request logging, or monitoring** of any kind.
- **No automated backups** configured for the database.
- **Not deployed anywhere** — runs locally only, against a local PostgreSQL instance.

## 11. Design Principles

These principles have guided decisions so far and should continue to:

1. **Free-first**: every tool, library, and service chosen has been free/open-source unless there's no reasonable alternative (see Section 12).
2. **API-first backend**: the backend has zero knowledge of the frontend's existence beyond CORS config. Any client (web, future mobile, CLI) could consume the same API.
3. **Thin routers, fat services**: business logic never lives in route handler bodies.
4. **Schema mirrors model mirrors service mirrors router**: one file per domain at each layer, named consistently, to keep the codebase predictable and easy to navigate as it grows.
5. **Compute, don't cache, nutrition totals**: correctness over performance at this stage — a deliberate, revisitable tradeoff (see Section 10).
6. **Ownership isolation by default**: any time a resource belongs to a specific dietitian (patients, meal plans), every query path checks ownership — there is no "trust the frontend" shortcut anywhere in the backend.
7. **Incremental, milestone-based development**: each milestone has shipped a vertically complete slice (DB → API → UI → verified), never just a backend or just a frontend in isolation.
8. **Migrations are law**: schema changes happen through Alembic only, never ad-hoc SQL or `create_all()`.

## 13. Things That Should Not Be Changed Without Discussion

The following decisions are foundational — changing them has ripple effects across multiple milestones already built. Do not change these unilaterally; raise it explicitly first:

- **JWT-based auth with access + refresh tokens stored in `localStorage`**. Switching to cookies/sessions, or to a third-party auth provider, affects every protected route on both frontend and backend.
- **The `model → schema → service → router` backend layering pattern.** Every existing module follows this. New modules should too.
- **UUID primary keys everywhere.** Switching any table to integer IDs breaks foreign key consistency project-wide.
- **The EAV-style `food_nutrients` table design.** This was a deliberate tradeoff to avoid schema churn as more nutrients get added. Don't "normalize it back" into fixed columns without understanding why it was built this way.
- **Nutrition totals computed at read-time, not stored.** Don't introduce a cached/stored totals column without also solving the staleness problem it would create.
- **Role names (`admin`, `dietitian`, `user`) and their Postgres enum representation.** Renaming or restructuring roles requires a migration and touches auth, dependencies, and every role-gated route/page.
- **The monorepo structure itself** (`frontend/`, `backend/`, `docs/`, `scripts/` at the root). Splitting into separate repos or rearranging top-level folders affects every relative import, every command in this documentation, and every future onboarding step.
- **Free-tooling constraint.** Don't introduce a paid dependency (hosting, database, auth, APIs) without first checking for a free-tier alternative and explicitly flagging the cost tradeoff.

## 14. Current Development Focus

**Active phase: UI/UX Overhaul**

The MVP feature set (Milestones 1–10) is functionally complete: auth, roles, admin panel, food database, food search, nutrient calculator, patient management, and meal planning all work end-to-end and have been manually verified.

The current focus is **not new features** — it's revisiting the existing screens with deliberate visual design rather than the functional-first Tailwind styling used to move quickly through the MVP. This includes (to be scoped in detail separately):
- Establishing a real design system/token vocabulary instead of organic Tailwind class repetition
- Improving responsive behavior (mobile/tablet has not been a consideration so far)
- Polishing loading states, empty states, and error states across all pages
- Visual hierarchy and information density review, especially on data-heavy screens (patient detail, meal plan builder)
- Accessibility pass (contrast, focus states, semantic markup)

No backend changes are expected during this phase. Any backend change discovered to be necessary during the UI work should be flagged before implementing, since it falls outside the current milestone's intent.
